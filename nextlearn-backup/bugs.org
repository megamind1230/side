* FIXED B25: Image centering clips bottom/right edges / can't pan to edge

** Location
- ~Views/MainWindow.axaml~ — ~Image~ element alignment attributes inside ~ScrollViewer~

** Status
- FIXED — set ~HorizontalAlignment="Left"~ and ~VerticalAlignment="Top"~ on Image

** Root Cause
~HorizontalAlignment="Center"~ / ~VerticalAlignment="Center"~ (or default ~Stretch~ with explicit Width/Height) causes the Image to be centered within the ScrollViewer's content area. The scroll range assumes content origin at (0,0), but centering shifts the image so that right/bottom edges extend past the scrollable range. Purely removing both attributes reverts to default ~Stretch~, which ALSO centers explicit-sized elements — same clipping.

** Fix
Set ~HorizontalAlignment="Left"~ and ~VerticalAlignment="Top"~ so the Image sits at (0,0) in the ScrollViewer content area. The scroll range [0, extent − viewport] correctly covers the full image. Visual centering is handled by scroll offset calculation in ~RecenterScrollViewerAfterLayout()~.

** Files changed
- ~Views/MainWindow.axaml~ — set ~HorizontalAlignment="Left"~ and ~VerticalAlignment="Top"~ on ~OverlayImage~

* FIXED B26: Zoom centering jumps when panned + two-sub-bug: initial center override

** Location
- ~Views/MainWindow.axaml.cs~ — ~OnScrollViewerScrollChanged()~, ~OnMainViewModelPropertyChanged()~

** Status
- FIXED

** Sub-bug A: preserve-center formula is viewport-agnostic
*** Root Cause
The preserve-center formula ~newOffset = oldOffset + (newExtent − oldExtent)/2~ only gives correct results when ~oldOffset = (extent − viewport)/2~ (image perfectly centered). For a panned image the formula fails because it ignores viewport size. Example: scrolled to Y offset 50 in a 1000×1000 extent with 600px viewport — old formula yields Y=300 on 1.5× zoom instead of correct Y=225 (75px jump).

*** Fix
Replaced with ~newOffset = (oldOffset + vp/2) * zoomRatio − vp/2~ which properly factors in viewport: the viewport-center point in content coords is ~oldOffset + vp/2~, scaled by zoom ratio, then back to offset.

** Sub-bug B: initial centering overridden to preserve on async bitmap load
*** Root Cause
When overlay opens, ~IsImageOverlayOpen~ handler calls ~RecenterScrollViewerAfterLayout(true)~, but ~UpdateOverlayImageSize()~ bails because ~Source~ is null (bitmap not loaded yet). When the async load completes, ~CurrentImageBitmap~ fires the shared zoom/rotation/bitmap handler with ~centerInViewport: false~ — overriding the intended center mode to preserve mode.

*** Fix
Separated ~CurrentImageBitmap~ from the ~ZoomLevel~/~RotationAngle~ handler. ~CurrentImageBitmap~ now uses ~centerInViewport: true~ so initial loads, next/previous images, and invert toggles all center correctly. ~ZoomLevel~ and ~RotationAngle~ continue to use preserve mode.

** Files changed
- ~Views/MainWindow.axaml.cs~ — fixed preserve formula in ~OnScrollViewerScrollChanged()~, split ~CurrentImageBitmap~ into its own handler with ~centerInViewport: true~

* FIXED B28: Org [[...]] file paths corrupted by italic regex

** Location
- ~Services/HtmlContentBuilder.cs~ — ~RenderOrgInline()~

** Status
- FIXED — placeholder protection added

** Root Cause
The italic regex ~/([^/]+)/~ (org-mode syntax) runs before org ~[[url]]~

placeholders AFTER formatting, before org link/image regexes run.

* B36: `o` key doesn't fire from WebView (tunneling vs bubbling)

** Location
- ~Views/MainWindow.axaml.cs~ — ~OnKeyDown()~, ~HandleKey()~, tunneling handler

** Status
- OPEN — needs a clean solution

** Root Cause
~KeyDown~ is a bubbling routed event. When the NativeWebView has focus and a
plain letter key (~o~) is pressed, the WebView consumes the event for text
input and never bubbles it to the MainWindow. ~?~ (Shift+Oem2) and ~Ctrl+,~
(Ctrl+OemComma) work because the WebView doesn't treat modifier-combination
keys as text input.

The tunneling workaround (~AddHandler(KeyDownEvent, handler, RoutingStrategies.Tunnel)~)
catches ~o~ before the WebView gets it, but requires an ~e.Source is TextBox~
guard to not steal input from real text boxes. This is fragile.

** Proper fix needed
NativeWebView should either:
1. Not consume plain letter keys in read-only mode (it's a display-only WebView)
2. Forward unhandled key events to the parent window

** Workaround applied
Tunneling handler in ~MainWindow()~ constructor, guarded by ~e.Source is TextBox~.

* FIXED B35: GLFW Wayland initialization failure on Hyprland

** Location
- ~Program.cs~ — ~BuildAvaloniaApp()~

** Status
- ✅ FIXED

** Root Cause
~Environment.SetEnvironmentVariable("WAYLAND_DISPLAY", "")~ calls
~setenv("WAYLAND_DISPLAY", "")~. GLFW's ~getenv("WAYLAND_DISPLAY")~ returns
a non-NULL pointer (empty string), so GLFW thinks Wayland IS configured
and tries to ~wl_display_connect("")~, which fails because the display
name is empty. This emitted ~10+~ "[glfw error 65544]: Wayland: Failed to
connect to display" messages on every startup.

** Fix
~Environment.SetEnvironmentVariable("WAYLAND_DISPLAY", null)~ calls
~unsetenv("WAYLAND_DISPLAY")~ → ~getenv("WAYLAND_DISPLAY")~ returns NULL →
GLFW skips the Wayland backend entirely and falls through to X11.
Combined with ~builder.UseSkia().UseX11()~ when ~XDG_SESSION_TYPE=wayland~.

** Files changed
- ~Program.cs~ — ~null~ instead of ~""~ for WAYLAND_DISPLAY env var

* CLOSED B42: Study page keyboard handling — layered fix needed

** Location

- ~NextLearn.Desktop/Views/MainWindow.axaml.cs~ (~HandleKey~ switch, line ~403~)
- ~NextLearn.Desktop/Services/KeyBindingService.cs~ (VimBindings missing ~S~ entry)

** Symptoms
1. C-c chords fail on study page (Ctrl+C intercepted before handler)
2. C-x chords fail on study page (Ctrl+X intercepted before handler)
3. C-v / M-v / C-b / C-f scroll keys don't work on study page
4. C-c s doesn't toggle sidebar from study page
5. C-x p/a/h doesn't open pinned/archived/heatmap from study page
6. C-c o/m doesn't open decks dir/marketplace from study page

** Root Cause
The issue had two layers, not three:

| Layer | Issue | Fix |
|-------|-------|-----|
| A | ~DecodeKeyUri~ only mapped 10 letters (n/p/j/k/h/l/q/d/e/i/g). All new JS-forwarded letters hit ~_ => Key.None~ and were silently dropped | Expand switch to all a-z, A-Z, ArrowUp, ArrowDown |

Layer C (WebKitGTK native keybinding signals) was hypothesized but turned out
to be a false alarm. The tunnel handler installed via
~AddHandler(InputElement.KeyDownEvent, ...)~ fires at the Avalonia event
tunnel phase — before the native WebKitGTK widget processes the key at the
GTK signal level. Avalonia's input system intercepts all keyboard events at
the window level first, then forwards them to the focused control. Since the
tunnel handler runs during this window-level pre-processing, it sees
EVERY keypress before WebKitGTK can consume it. This is inherent to
Avalonia's event routing architecture — no GTK EventControllerKey hack is
needed.
